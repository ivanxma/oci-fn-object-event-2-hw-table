Large-file CSV import resolution plan

Finding: the current Object Storage Event → synchronous OCI Function path cannot reliably import very large files. The 1 GB test reached the OCI maximum synchronous timeout of approximately 300 seconds, including with 16 writer workers. The file was streamed successfully, but the invocation terminated before completion.

1. Target architecture

StageDesign
Event intakeObject Storage Event Rule invokes a short handler. The handler validates the mapping, records the event and object ETag, and returns quickly.
Work orchestrationPublish a durable work message containing bucket, object name, version/ETag, mapping ID, target table, and correlation ID to OCI Queue or Streaming.
Long-running workerConsume the message and process the object in detached mode or in multiple bounded chunks. Use Object Storage range reads, streaming CSV parsing, staging tables, and partition exchange.
CompletionWrite phase timings and terminal state to the control schema. Acknowledge the queue message only after success; retry with backoff on transient errors.

2. Preferred options

OptionUse whenTrade-off
Detached Function workerA single object normally completes within one hour.Detached execution supports up to 3,600 seconds, but requires a short synchronous submitter and idempotent worker state.
File chunkingFiles may exceed one-hour processing time or need parallelism.Split the source into independent CSV objects, each below a safe duration budget (for example 100–250 MB). Each chunk maps to a staging partition and a manifest tracks all chunks.
OCI Data Integration / MySQL Shell jobBulk loads are scheduled or operationally managed rather than event-latency sensitive.Long-running job control is stronger, but application-level event integration is less direct.

3. Detached worker workflow

  1. Event handler persists RECEIVED with object identity and ETag.
  2. Handler submits a detached worker request and records DETACHED_SUBMITTED.
  3. Worker acquires a lease keyed by object version; duplicate events exit safely.
  4. Worker streams ranges, validates the header, loads staging, and records phase timings.
  5. Worker exchanges the staging table with the target partition in one transaction.
  6. Worker records SUCCESS or ERROR, including retry count and diagnostic text.

4. Chunked workflow

  1. A producer creates a manifest containing chunk names, byte/row ranges, checksums, and expected row counts.
  2. Each chunk is uploaded under a unique prefix and produces its own event.
  3. Workers process chunks independently with 4–16 writer threads, subject to database I/O limits.
  4. A coordinator marks the logical file complete only when every chunk succeeds and all checksums/row counts match.
  5. Partition exchange occurs per chunk or in a final controlled merge, depending on transaction requirements.

6. Reliability and transaction requirements

  • Use object ETag/version ID for idempotency; never process the same version twice.
  • Keep staging and target updates atomic per chunk or partition exchange.
  • Do not claim logical-file success until every chunk is successful.
  • Retain raw event, worker, phase, retry, and terminal records for audit.
  • Use dead-letter handling for permanently failing messages.

7. Dynamic mapping-driven execution mode

Resource Mapping becomes the runtime association between Object Storage objects and target tables. Add invocation_mode (SYNC or DETACHED), worker_threads, batch_rows, timeout_seconds, and enabled to each mapping.

Mapping modeFunction behaviorLimit
SYNCResolve the mapping and process in the current Event Rule invocation.Maximum 300 seconds.
DETACHEDPersist the event, submit a detached worker with the mapping snapshot, and return quickly. The worker performs the import.Detached execution supports up to 3,600 seconds; requires self-invoke IAM permission.

The OCI Event Rule itself does not dynamically switch invocation type. Dynamic behavior is implemented by the Function handler after it resolves the mapping. New events use the latest mapping; in-flight workers retain a snapshot of the mapping used to start them. The UI should expose mode, timeout, worker count, and state.

8. UI and operational visibility

The UI must record and display the execution mode selected for every event. Add invocation_mode, mapping ID, worker count, timeout, detached invocation ID, parent event ID, queue message ID, submitted time, started time, completed time, duration, retry count, and terminal status to the event/audit records.

Add a dedicated Detached Processes tab alongside Object Storage Events and Event TX. It should provide filters for state, mapping, target table, object name, date range, and correlation ID, with refresh/polling and links to the original event, detached invocation, logs, staging table, and terminal transaction.

Detached tab viewRequired information
SummaryReceived, submitted, processing, success, error, timeout, and retry counts.
Process detailObject, ETag/version, mapping, target table, mode, worker count, phase timings, and error message.
ActionsRetry failed work, cancel where supported, inspect logs, and clean residual staging resources.

9. Rollout plan

  1. Implement the short event handler and control-state transitions.
  2. Add detached worker invocation and verify a 500 MB object.
  3. Add Queue/Streaming retry and dead-letter handling.
  4. Introduce chunk manifests and test a 1 GB logical file.
  5. Run 100 MB–1 GB matrices on MySQL.2 and MySQL.8 while collecting CPU, IOPS, throughput, queue depth, and latency.
  6. Promote only after integrity, retry, and recovery tests pass.
Recommendation: adopt the short event-handler + durable queue + detached/chunked worker architecture. It removes the 300-second synchronous ceiling while preserving event-driven processing, transaction boundaries, and complete operational visibility.
