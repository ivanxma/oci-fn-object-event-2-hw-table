Ordered Object Event Queue and OCI Function Revision Plan

Project: OCI Object Event to HeatWave Table  |  Plan date: 19 July 2026  |  Target stack: Flask, Python 3.13+, Oracle MySQL Connector/Python 9.7+

Problem statement. Invocation mode does not guarantee event order. A detached CREATE may still be loading when a later DELETE runs synchronously or in another detached invocation. The DELETE can finish first and the older CREATE can then publish data that should remain deleted. OCI Events delivery, Function concurrency, and detached self-invocation must therefore be treated as unordered and at-least-once inputs.

1. Required outcome

Introduce a durable MySQL-backed work queue between event intake and data mutation. The queue serializes operations that affect the same target-table mapping, retains every attempt for audit, and lets one detached Function invocation process multiple entries while it has enough safe runtime remaining. The UI provides an operational dashboard and controlled create, edit, cancel, retry, and cleanup actions.

  • Ordering: CREATE, UPDATE, and DELETE execute in effective event order within a queue lane.
  • Single owner: only one worker can mutate a lane at a time, even if several detached invocations are submitted.
  • Durability: receiving an event and inserting its queue entry is one database transaction.
  • Time safety: a worker claims another item only when the estimated work fits before its soft deadline.
  • Recovery: leases expire, retries are bounded, and poisoned entries move to a dead-letter state without silently blocking operations.

2. Target architecture

Queue binding: per table or per mapping

Each resource mapping must define a queue_scope of TABLE or MAPPING. The Function calculates and snapshots a stable queue_binding_key when it enqueues the event. This makes queue ownership explicit instead of deriving it differently in the UI and worker.

Queue scopeBinding keyUse and safety rule
TABLE (safe default)table:{target_database}.{target_table}All mappings that write the same target table share one ordered lane. Use when files can interact through keys, row moves, partition operations, or create/delete ordering.
MAPPINGmapping:{mapping_id}Only events for that mapping are serialized. Allow this when the mapping owns an independent, non-overlapping partition and cannot conflict with another mapping targeting the same table.
Validation rule. If more than one enabled mapping targets the same table, the UI defaults them to TABLE and warns before permitting MAPPING. All mappings sharing a table-scoped queue resolve to the same binding key. Changing queue scope is blocked while either the old or new binding has non-terminal work, unless an administrator performs an audited migration.

Queue states: PENDING → LEASED → RUNNING → SUCCESS. Alternate states are RETRY_WAIT, BLOCKED, CANCELLED, and DEAD_LETTER. Rows are retained; UI “delete” means audited cancellation/tombstoning unless a retention job later purges old terminal history.

4. Ordering and transaction rules

  1. The intake handler resolves and snapshots the mapping, calculates its table- or mapping-scoped binding key, inserts the raw event and queue row in one transaction, then requests detached dispatch and returns.
  2. Within a lane, select the first eligible row by event_time ASC, received_at ASC, id ASC. The database ID is the deterministic final tie-breaker.
  3. Use a short configurable reorder grace period before an entry becomes eligible, allowing a slightly delayed earlier event to arrive. Event time is not, by itself, a guaranteed sequence number.
  4. Lock the lane and queue row using an atomic conditional update or SELECT ... FOR UPDATE SKIP LOCKED. Commit the claim before beginning streaming work.
  5. Never skip an earlier non-terminal entry in the same lane. A terminal error blocks later entries until policy retries it, an operator cancels it, or an operator explicitly advances the lane with an audited override.
  6. Before partition exchange or DELETE commit, compare the claimed entry with the lane generation and latest known object version. Abort stale work rather than publish an older state.
  7. Only after the target mutation and transaction log are committed does the worker mark the entry SUCCESS and advance the lane watermark.
Important limitation. Object Storage event timestamps and delivery order are not a complete business transaction protocol. If strict cross-file ordering is mandatory, producers should supply a manifest or monotonically increasing sequence. Without that, the queue can provide deterministic observed-event order plus a grace window, but it cannot know about an event that has not yet arrived.

5. Function control flow

Runtime budget policy

Record a monotonic invocation start time and calculate soft_deadline = configured_detached_timeout - shutdown_margin. The configured detached timeout cannot exceed 3,600 seconds. Before claiming the next entry, require:

remaining_seconds > max(minimum_start_budget, predicted_p95_seconds × safety_factor) + commit_reserve_seconds

SettingInitial valueReason
Detached Function timeout3,600 sOCI detached execution ceiling used by this design.
Shutdown/commit reserve120 sLeaves time to rollback, record state, release the lease, and request continuation.
Minimum start budget180 sAvoids starting even a small unknown job close to timeout.
Prediction safety factor1.35Uses object bytes and rolling p95 throughput for the same mapping/shape/worker count.
Lane lease60 s, renewed every 20 sFast recovery without allowing concurrent owners; actual values require fault testing.
Reorder grace15–60 s, configurableReduces late-arrival reordering; strict producers should use manifest sequence numbers.

If one entry is predicted to exceed the usable detached budget, do not claim it repeatedly. Mark it BLOCKED with reason REQUIRES_CHUNKING_OR_EXTERNAL_JOB, and expose remediation in the UI.

Handling several detached invocations

It is acceptable for every intake event to request detached execution. The lane lease makes redundant invocations cheap: only the owner processes work; others observe the live lease and exit successfully. To close the enqueue/worker-exit race, maintain an atomic dispatch_requested flag and generation counter. When an owner cannot start the next item, it creates at most one continuation request, commits that dispatch generation, then releases its lease. A newly arriving intake either observes an owner or becomes responsible for dispatch—never neither.

6. Failure, retry, and idempotency behavior

ConditionRequired behavior
Duplicate OCI eventUnique idempotency key returns the existing queue entry; no duplicate target mutation.
Detached invoke 429/5xxRecord dispatch failure and retry with bounded exponential backoff and jitter. Because invocation outcome can be ambiguous, lane leasing remains mandatory.
Worker crash/timeoutLease expires; a recovery invocation requeues or resumes according to the last durable phase. Uncommitted MySQL work rolls back.
CSV/schema/data errorMove to BLOCKED or DEAD_LETTER after configured attempts; preserve staging diagnostics and block later lane work until resolved.
DELETE after CREATEDELETE cannot be claimed until CREATE is terminal. A stale CREATE must recheck the lane generation immediately before publish.
Manual cancellationPending work becomes CANCELLED. Running work receives a cancel request and stops only at a safe checkpoint; it is never physically deleted mid-transaction.

8. Function and UI implementation changes

  1. Schema migration: add mapping queue_scope plus queue, lane, attempt, and transition-audit tables with indexes for binding/status/order and immutable idempotency constraints.
  2. Event intake: revise function/func.py so both SYNC and DETACHED mappings persist queue work first. Mapping mode becomes a dispatch policy, not permission to bypass ordering.
  3. Queue service: create a shared Connector/Python module for transactional enqueue, lane election, claim, heartbeat, completion, retry, continuation, and recovery.
  4. Worker loop: wrap the existing streaming CREATE/UPDATE and DELETE operations with claims, generation checks, runtime budgeting, and append-only attempts.
  5. UI: add queue scope to mapping create/edit/detail and add Queue blueprint/service/templates, dashboard, CRUD-style controlled operations, binding detail, filters, paging, refresh, and export using existing shared components.
  6. Deployment: migrate the control schema idempotently; dynamically discover the Function OCID as today; deploy the timeout, memory, concurrency, worker-count, lease, grace, retry, and safety-margin settings.
  7. Compatibility: retain current Object Storage Event, Event TX, Detached Process, and Registered Table views, linking them to the new queue entry and attempt IDs.

10. Test matrix

  • CREATE then DELETE while CREATE is still streaming; UPDATE then DELETE; DELETE then later CREATE.
  • Ten duplicate deliveries of the same event; two invocations racing for one binding; separate tables running concurrently.
  • Two mappings targeting one table under TABLE scope serialize together; the same mappings under an explicitly approved MAPPING scope use independent lanes.
  • Late earlier event inside and outside the reorder grace period; same timestamps with deterministic ID tie-break.
  • Worker killed during stream, staging write, exchange, audit commit, and continuation dispatch.
  • 503 detached dispatch, expired lease, heartbeat loss, dead-letter, manual retry, cancel, and audited lane advance.
  • Worker receives two small entries and drains both; insufficient remaining budget leaves the second pending and launches exactly one continuation.
  • One entry estimated beyond 3,600 seconds is blocked for chunking instead of repeatedly timing out.
  • Flask authenticated route/template/JavaScript tests, CSRF/authorization tests, SQL identifier validation, secret scan, and relevant Python compilation under Python 3.13+.

11. Assumptions and limitations

  • At-least-once delivery and duplicate detached invocations are normal; correctness comes from queue idempotency and leasing.
  • A mapped file is a non-overlapping partition of a table. Duplicate records across files remain unsupported.
  • Moving rows between files is not a multi-file atomic operation. Where business semantics require removal before addition, producers should express that order through a manifest/sequence.
  • Queue ordering cannot reconstruct an event that never arrives or determine producer intent from timestamps alone.
  • Atomicity remains per queue entry/partition exchange. A multi-file manifest requires an additional coordinator if all files must commit as one logical unit.
  • Files that cannot safely complete within detached runtime must be split into ordered chunks or moved to a long-running external job platform.
  • Manual queue operations are privileged recovery tools and can affect data integrity; they require reason text, confirmation, and audit.
Recommended implementation decision. Enqueue every mapped event and use one leased worker per explicit queue binding. Default the binding to the target table; allow per-mapping binding only for validated independent data partitions. Let the owner drain entries only while predicted work fits inside a soft deadline. Redundant detached executions become harmless wake-up attempts, while the durable queue—not Function arrival order—controls the final database state.
