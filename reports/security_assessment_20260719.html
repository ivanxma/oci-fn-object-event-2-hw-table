Executive summary

Overall risk: High; immediate containment required. The system has sound foundations for data validation, secret-file permissions, server-side credential state, and OCI resource principals. However, one Critical profile-management flaw can let an unauthenticated party replace a trusted connection profile and redirect a later operator login to an attacker-controlled MySQL endpoint. Privileged OCI operations also rely on authentication alone, without role-based authorization.

Detailed findings

SEC-01 — Public connection-profile creation permits trusted-profile replacement

Critical
Affected
profile_routes.py, profile_store.py, profile_form.html
OWASP
A01 Broken Access Control; A07 Identification and Authentication Failures; A10 SSRF
Confidence
Confirmed by source and deployed behavior

/profiles/new intentionally has no authentication guard. ProfileStore.save() removes a profile with the submitted name and replaces it, even on this public create path. An attacker can replace a trusted profile's host with an endpoint they control. The next legitimate login sends the supplied MySQL username/password to that endpoint. The route also accepts SSH private-key uploads before authentication, allowing repeated disk consumption, and arbitrary host/port entries create an SSRF/port-probing path when a login is attempted.

GET /profiles/new → HTTP 200 without authentication POST /profiles/new without a token reached form processing → HTTP 400 (not 401/403) profile_routes.py: create() has no @login_required profile_store.py: save() removes existing entries matching the submitted name before append

Remediation: remove public profile mutation. Bootstrap profiles through deployment or a one-time, localhost-only setup token. Require an authenticated, explicitly authorized administrator for create/edit/delete; reject create when the name exists; separate create and update semantics; require CSRF; restrict destination hosts/ports to approved networks; rate-limit uploads; enforce a much smaller SSH-key size; log profile changes without logging sensitive values. Immediately verify deployed profile hosts through a trusted channel and rotate operator DB credentials if unauthorized modification cannot be excluded.

SEC-02 — Authentication is used as authorization for privileged operations

High
Affected
Mapping, OCI Rule, Function Configuration, Object Storage, staging cleanup, import, and profile routes
OWASP
A01 Broken Access Control
Confidence
Confirmed

Every authenticated database user receives the same application privileges. Routes that update OCI Function memory/timeouts, disable/delete Events rules, upload/delete bucket objects, drop staging tables, create schemas, and load data use only @login_required. On the deployed VM both OCI management flags are enabled, so the UI instance principal's cloud privileges are reachable through any valid configured MySQL login.

OCI_FUNCTION_CONFIGURATION_ENABLED=true OCI_EVENT_RULE_MANAGEMENT_ENABLED=true No role/permission decorator or admin-profile check was found Deployment identity is expected to manage functions-family, cloudevents-rules, logging, and test objects

Remediation: define roles such as Viewer, Data Operator, Mapping Administrator, and Platform Administrator. Resolve identity independently from arbitrary MySQL credentials, enforce authorization on every GET and POST route, and apply least-privilege OCI policies to separate UI control-plane functions. Require step-up authentication for Function configuration, rule deletion, bulk cleanup, and object deletion. Disable control features at deployment when not required.

SEC-03 — State-changing forms do not enforce CSRF tokens

High
Affected
15 POST-form groups across login, logout, profiles, mappings, OCI rules, uploads, imports, and cleanup
OWASP
A01 Broken Access Control
Confidence
Confirmed

No CSRF library, token, synchronizer pattern, or Origin/Referer validation is implemented. A POST to /logout with an authenticated cookie and no token returned 302, demonstrating that state-changing requests do not require proof of page origin. SameSite=Lax is a useful mitigation for common cross-site form submissions, but it is not a complete CSRF control for same-site subdomains, browser variations, or chained vulnerabilities.

Source search for csrf/xsrf → no implementation Authenticated POST /logout without token → HTTP 302 Cookie: Secure; HttpOnly; SameSite=Lax

Remediation: add a maintained CSRF implementation such as Flask-WTF CSRFProtect; place tokens in all forms and AJAX requests; reject missing/invalid tokens; verify Origin for high-risk endpoints; use POST/DELETE semantics consistently; keep SameSite and explicit confirmations as secondary controls. Add negative tests for every state-changing route.

SEC-04 — TLS and SSH protect encryption but do not verify peer identity

High
Affected
nginx TLS certificate, mysql_service.py, Function database config, ssh_tunnel.py
OWASP
A02 Cryptographic Failures
Confidence
Confirmed

The deployed HTTPS certificate is self-signed and its CN is 145.241.232.76, not the assessed endpoint 150.230.121.100. Operators must bypass a browser warning, defeating server authentication. Connector/Python forces TLS but does not set ssl_verify_cert or ssl_verify_identity; Oracle documents both defaults as false. SSH tunnels do not supply a pinned host key or known-hosts policy. The Function similarly supports disabling DB TLS and has no CA/identity settings.

Certificate subject/issuer: CN=145.241.232.76 (self-signed) Certificate validity: 2026-07-19 through 2027-07-19 MySQL args: ssl_disabled=False and ClientFlag.SSL, no ssl_ca/ssl_verify_cert/ssl_verify_identity SSHTunnelForwarder: no ssh_host_key parameter

Remediation: deploy a CA-issued certificate matching the production DNS name; do not train operators to bypass warnings. Configure MySQL VERIFY_IDENTITY with an approved CA and hostname, including the Function. Pin or verify SSH host keys from a protected known-hosts store. Remove or tightly restrict any TLS-disable override. Add automated certificate-expiry and hostname checks.

SEC-05 — Login attempts and credential-bearing sessions have no bounded lifetime

Medium

Six consecutive invalid-profile login requests all returned HTTP 200; no throttling, delay, lockout, or alerting was observed. SessionStore retains plaintext DB credentials in process memory until explicit logout or a later health-check failure. Sessions have no idle/absolute expiry, no maximum count, and no janitor. Closing a browser leaves server-side credentials and uploaded import jobs resident until restart.

Remediation: add IP/account/profile-aware throttling at nginx and Flask; use exponential delay and alerts; set idle and absolute session limits; expire and clear server entries independently of browser activity; cap active sessions/import jobs; rotate the session identifier after login; document that a UI restart invalidates sessions. Consider short-lived delegated DB credentials instead of reusable passwords.

SEC-06 — Browser security headers are absent

Medium

The HTTPS response did not include HSTS, Content-Security-Policy, frame protection, X-Content-Type-Options, Referrer-Policy, or Permissions-Policy. The response also discloses nginx/1.20.1. Flask recommends reviewing these headers; CSP and frame restrictions are particularly important because this UI performs destructive administrative actions.

Remediation: after installing a valid certificate, enable HSTS; introduce a nonce-based CSP and move inline scripts/handlers into static files; set frame-ancestors 'none', X-Content-Type-Options: nosniff, a restrictive Referrer-Policy and Permissions-Policy; disable nginx version tokens. Test every page before enforcing CSP.

SEC-07 — Production deployment uses Flask's development server and limited sandboxing

Medium

The systemd unit starts python -m flask ... run. Flask emits a production warning. The container correctly runs as appuser and is not privileged, but uses host networking, a writable root filesystem, and no explicit capability drop or no-new-privileges setting. systemd-analyze security rated the unit 9.2 UNSAFE; this heuristic includes the privileged Podman launcher but still identifies meaningful missing hardening.

Container: User=appuser; Privileged=false; ReadOnlyRootfs=false; NetworkMode=host Listener: Flask only on 127.0.0.1:8080; nginx on 0.0.0.0:443 systemd-analyze security exposure: 9.2 UNSAFE

Remediation: use Gunicorn or another supported production WSGI server with bounded workers/timeouts; use a dedicated bridge or slirp network; drop all capabilities, add only those proven necessary, set no-new-privileges, read-only rootfs with explicit writable mounts, resource limits, and a health check. Strengthen the systemd unit with an evaluated subset of ProtectSystem, ProtectHome, PrivateDevices, RestrictAddressFamilies, LockPersonality, and MemoryDenyWriteExecute.

SEC-08 — Upload validation and lifecycle controls are incomplete

Medium

Positive controls include a global 25 MB request cap, generated local filenames, secure_filename, CSV extension checks, folder traversal rejection, and storage outside the web root. However, extension alone does not establish CSV content; Object Storage testing streams the file without parsing, and local import validation samples only a bounded preview before later reading all rows. There are no per-user quotas, field/row limits, malware/content policy, rate limits, or independent cleanup schedule for abandoned uploads.

Remediation: validate structure and UTF-8 before publishing, enforce field length/column/row limits, reject NUL/control abuse, apply per-user and global quotas, and add a periodic janitor based on server-owned job timestamps. Keep generated names and out-of-webroot storage. Treat CSV formula injection when exporting or when downstream users open source data in spreadsheets.

SEC-09 — Long-lived secrets are supplied through environment and Function configuration

Medium

deploy/env.sh and the UI runtime environment are correctly mode 0600 and untracked. However, deployment sends DB_PASSWORD into OCI Function configuration and keeps OCIR/DB credentials in the deployment environment. Any identity allowed to inspect those resources may gain access. Rotation requires redeployment and is not centrally governed.

Remediation: store database credentials in OCI Vault, grant the Function resource principal permission to read only the intended secret, retrieve and cache it for a bounded period, and implement rotation. Prefer workload identity/token login where supported. Keep the deployment file only for bootstrap references and never return secret values through the Function Configuration UI.

SEC-10 — Builds resolve broad dependency ranges without locks, hashes, or attestations

Medium

Requirements use broad ranges and the Docker base is a mutable tag. Rebuilding can silently select different transitive packages. No lock file, hashes, SBOM, image signature, provenance attestation, or CI security gate is defined. This is a reproducibility and supply-chain issue even though the current image scan was favorable.

Remediation: generate reviewed lock files with hashes for Python 3.13, pin the base image by digest, produce CycloneDX/SPDX SBOMs, scan in CI, sign images, enforce signature verification on deployment, and schedule rebuilds for patched base layers. Maintain compatible upper bounds, but update through controlled automation.

SEC-11 — Paramiko 3.5.1 is affected by CVE-2026-44405

Low

pip-audit resolved paramiko==3.5.1 from the declared >=3.4,<4 constraint and reported PYSEC-2026-2858 / GHSA-r374-rxx8-8654 / CVE-2026-44405. The reviewed advisory rates it Low (CVSS 3.4) and describes acceptance/use of SHA-1 in RSA key handling through version 4.0.0 before a specific commit. The current constraint excludes Paramiko 5.0.0.

Remediation: compatibility-test sshtunnel and this app with Paramiko 5, then update the constraint and rebuild. If SSH profiles are not required, remove both optional SSH dependencies and disable that code path.

SEC-12 — Operational payloads and errors lack explicit redaction/retention policy

Low

The control schema preserves raw CloudEvent JSON, object names, bucket/compartment identifiers, database errors, and transaction messages. All authenticated users can inspect Event TX. Exception traces are logged for many OCI failures and detailed SDK messages are sometimes flashed to the UI. These records are useful, but no retention, sensitivity classification, redaction, or role-based visibility policy is defined.

Remediation: define retention and purge schedules, redact tokens/query strings and unexpected payload fields, show correlation IDs plus safe guidance in the UI, retain detailed diagnostics in a restricted log, and restrict raw-event/error views to support roles. Protect database audit tables from modification by the loader/UI accounts where feasible.

SEC-13 — Suppressed exceptions can hide evidence

Low

Bandit identified several except Exception: pass paths around batch error marking, staging cleanup, and error logging. Some are deliberate to preserve the primary failure, but completely suppressing the secondary exception makes forensic reconstruction and orphan cleanup less reliable.

Remediation: emit a structured warning with correlation/event ID and exception class, without credentials or raw secrets. Track cleanup-required state explicitly and alarm on repeated audit-write failure.

Verified controls and favorable results

  • Browser cookie contains only an opaque, 256-bit server-session identifier.
  • Deployed cookie is Secure, HttpOnly, and SameSite=Lax.
  • Database passwords remain in server memory, not Jinja templates or profile JSON.
  • Profiles file is 0600; instance/key directories are 0700.
  • env.sh and UI runtime environment are 0600 and not Git-tracked.
  • Authenticated dashboard responses set no-store/no-cache directives.
  • Jinja autoescaping is retained; no unsafe |safe or dynamic template rendering was found.
  • SQL values are parameterized and dynamic identifiers pass strict allowlist validation.
  • Object prefixes reject traversal and selected deletions must match the mapping pattern.
  • Upload request size is capped at 25 MB and local names include a generated UUID.
  • MySQL transactions roll back on failure and connections close in finally.
  • Function and UI OCI access use resource/instance principals rather than checked-in API keys.
  • UI container runs as non-root, is unprivileged, and Flask binds only to loopback behind nginx.

Required security regression tests

  1. Unauthenticated users cannot create, replace, enumerate, edit, or delete profiles and cannot upload SSH keys.
  2. A Viewer cannot call any mapping, rule, Function, object-delete, cleanup, profile, or import mutation endpoint; an authorized role can perform only its assigned actions.
  3. Every state-changing endpoint rejects a missing, invalid, replayed, or cross-session CSRF token.
  4. Login throttling works by source and account/profile without disclosing whether a profile or username exists.
  5. Signed cookies contain no username, password, profile host, OCI identifier, or server-side credential state.
  6. Idle and absolute expiry remove the browser session, server credential record, tunnel, and temporary uploads.
  7. MySQL VERIFY_IDENTITY accepts the approved endpoint and rejects an untrusted CA or wrong hostname; SSH rejects unknown/changed host keys.
  8. TLS hostname, trust chain, expiry, HSTS, CSP, clickjacking, nosniff, referrer, and permissions headers pass automated checks.
  9. Upload tests cover oversized requests, long rows/fields, too many columns/rows, invalid UTF-8, control characters, duplicate/traversal names, quota exhaustion, and janitor cleanup.
  10. OCI policies demonstrate least privilege for UI and Function principals; detached self-invocation is limited to the intended Function.
  11. Container scan, pip-audit, Bandit triage, secret scan, SBOM generation, signature verification, Python compilation, shell syntax, and Flask route tests run in CI.
