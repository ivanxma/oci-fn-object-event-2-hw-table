Detailed CSV to HeatWave performance report

Executive summary

The final 500 MB comparison contains three successful runs: MySQL.2 with 4 workers completed in 240.172 seconds; MySQL.8 with 4 workers completed in 180.683 seconds; and MySQL.8 with 16 workers completed in 143.708 seconds (Event TX duration 142.962 seconds). Increasing workers helped on MySQL.8, while the monitoring data indicates storage I/O remained an important constraint.

The 1 GB object uploaded successfully but timed out at the synchronous OCI Function limit of 300 seconds. It is not a failed upload; it is a processing-timeout result.

2. Data and execution method

Files were generated on the VM with deterministic rows using the target schema:

record_id,event_ts,category,payload,amount

event_ts uses MySQL DATETIME syntax (YYYY-MM-DD HH:MM:SS). Objects were uploaded with OCI CLI. The Function streamed Object Storage ranges directly; no temporary CSV was created inside the Function container.

3. Timing definitions

  • Upload duration: wall-clock interval around oci os object put on the VM.
  • Function duration: OCI invocation log interval from “Received function invocation request” to “Served function invocation request”.
  • End-to-end latency: upload completion to terminal Event TX success; this requires correlating the upload timestamp with the control-schema terminal record.

6. Long-running execution limitation

Object Storage Events invoke the FAAS action synchronously. The Function cannot run beyond OCI's timeoutInSeconds=300 limit; the 1 GB test reached FunctionInvokeTimeout (504) at approximately 300 seconds.

Detached invocation supports up to 3,600 seconds, but is not selected directly by the current Event Rule. A future design needs a short Event Rule handler that persists the event and submits a detached worker, or routes through Queue/Streaming. The worker carries object identity, ETag, mapping ID, and correlation ID, with states RECEIVED → DETACHED_SUBMITTED → PROCESSING → SUCCESS/ERROR.

7. Monitoring interpretation: MySQL.8 storage bottleneck

The MySQL.8 retest used a 50 GB disk configuration with only one provisioned LUN and a maximum of approximately 3,750 IOPS. This storage ceiling is a likely bottleneck for staging-table writes, CSV validation, and partition exchange. The larger MySQL.8 shape did not translate directly into higher ingestion throughput because the disk path remained constrained.

ShapeObserved CPUInterpretation
MySQL.2Up to 100%CPU-bound during ingestion; additional compute may improve throughput.
MySQL.8Approximately 30%CPU underutilized while the single-LUN/3,750-IOPS storage path likely limits progress.

Follow-up testing should monitor disk IOPS, throughput, queue depth, and latency alongside CPU. Increase storage performance (more provisioned IOPS and/or additional LUNs with appropriate striping) before comparing compute shapes. Record the exact disk configuration and ensure the target and staging paths use the intended storage volume.

9. Disk write-operation evidence

The monitoring series for DbVolumeWriteOperations is mostly stable around 8–10 operations per minute, with brief elevated values at 16:27 (17.587), 16:28 (11.788), 16:31 (13.794), and major spikes at 16:32 (195.704), 16:33 (496.508), and 16:34 (330.222). After 16:35 the metric returns near the 8–10 baseline.

This evidence supports the storage-bottleneck hypothesis for MySQL.8: CPU can remain near 30% while the single-LUN volume absorbs burst write pressure. Correlate these timestamps with Function phases, volume throughput, IOPS, queue depth, and latency before increasing worker count further.

10. Observations

  • The 500 MB run completed in approximately four minutes and used 32 MiB Object Storage range responses.
  • Small files are dominated by fixed Event Rule/Function startup overhead; larger files expose streaming and database load throughput.
  • The 1 GB case is intentionally detached because generation and upload exceed an interactive terminal window.
  • Do not project p95/p99 or sustainable throughput from this sequential sample; concurrency and repeated trials remain outstanding.
Next steps: wait for the 1 GB marker, collect its Function log, then run fan-out (10–100 small files) and mixed create/update/delete cases. Capture terminal Event TX timestamps and database row counts for the final capacity report.
