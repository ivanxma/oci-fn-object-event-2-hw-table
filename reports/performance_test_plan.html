CSV to HeatWave performance test plan

A repeatable plan for measuring OCI Object Storage event ingestion through OCI Functions, staging tables, partition exchange, and the MySQL/HeatWave publication path.

Scope: file size, file count, target-table fan-out, lifecycle-event mix, capacity projection, and operational limits.

1. Purpose and test questions

File-size scalingHow do elapsed time, throughput, and temporary storage change from 10 KB through 10 MB, then from 100 MB through 1 GB?
Event-rate scalingHow many small-file events can be accepted and completed per minute when 10–100 files are routed to different target tables?
Lifecycle behaviourHow do create, update, and delete events affect latency, contention, correctness, and cleanup?
Capacity decisionWhat Function concurrency, database capacity, temporary storage, and operational guardrails are needed for the expected production workload?
Primary outcome: publish a measured operating envelope, not a single best throughput number. Report the sustainable rate where p95 latency, error rate, database load, and temporary-storage headroom remain within agreed limits.

2. System under test and timing model

3. Preconditions and assumptions

  • Use a dedicated bucket prefix, test schemas, and test mappings. Never use production target tables for load testing.
  • Each CSV is a complete, disjoint logical partition of its target table. Do not distribute the same business key across active files.
  • Use production-like column count, row width, character set, type conversion, indexes, partition layout, Function memory/timeout, network path, and HeatWave publication policy.
  • Record the exact deployed Function image, UI version, MySQL/HeatWave version, OCI region, Function shape/memory, writer-worker setting, and batch-row setting for every run.
  • Run a baseline with no competing workload, then repeat selected scenarios with representative analytical queries and controlled database background activity.
  • Warm up each configuration with at least three discarded runs. Execute at least five measured repetitions per data point; report median, p95, minimum, and maximum rather than only an average.

4. Prepare the test machine, OCI resources, and deployment

Use a dedicated test machine with the OCI CLI, Git, Python 3.13+, and MySQL client installed. Its clock, the Function environment, and MySQL should use NTP; record UTC timestamps everywhere. Do not put credentials in the run manifest or final report.

TEST_ROOT="$HOME/oci-object-event-performance"
RUN_ID="run-$(date -u +%Y%m%dT%H%M%SZ)"
mkdir -p "$TEST_ROOT/$RUN_ID"/{input,manifest,results,logs,sql}
chmod 700 "$TEST_ROOT" "$TEST_ROOT/$RUN_ID"

# Use the protected deployment configuration already maintained on the test machine.
# Do not copy it into Git, result bundles, or browser-visible files.
set -a
. /path/to/oci-object-event-2-table/deploy/env.sh
set +a
export PERF_BUCKET="$OBJECT_STORAGE_BUCKET_NAME"
export PERF_PREFIX="performance/$RUN_ID"
  1. Create a separate Object Storage bucket or, at minimum, a unique $PERF_PREFIX/ prefix. Enable bucket object events and retain the bucket name, namespace, compartment display name, and region in the run configuration.
  2. Deploy an isolated test Function application where possible. In its protected env.sh, set OBJECT_STORAGE_BUCKET_NAME to the test bucket and OBJECT_STORAGE_OBJECT_NAME_PATTERN to performance/*/*.csv; use test database credentials and a dedicated Function log group.
  3. Run deploy/deploy.sh from the test machine. It creates or updates the Function and OCI Events rule for create, update, and delete events. Confirm the bucket has object events enabled and record the rule and Function OCIDs.
  4. Verify the test Function resource principal can read only the test bucket, and that the Function subnet can reach the test MySQL endpoint. Keep deployment identity, Function resource-principal, and human/test-machine credentials separate.
  5. Before each test run, save a redacted configuration snapshot: Function image/version, memory/timeout, BATCH_ROWS, WRITER_WORKERS, Event Rule filters, database version, and HeatWave state.
Isolation rule: a performance run must have its own bucket prefix, mappings, tables, and Function/Event Rule target. Never point a test rule at a production bucket or a production target table.

5. Create existing test tables and resource mappings

The loader expects a pre-existing target table. Create one test table per intended target in the fan-out scenario; the table below is a representative template. Adjust business columns and indexes to match the real workload, but retain the invisible batch_num, LIST partitioning, and unique-key rule.

CREATE DATABASE IF NOT EXISTS perf_hw;

CREATE TABLE perf_hw.perf_t_001 (
  record_id BIGINT NOT NULL,
  event_ts DATETIME(6) NOT NULL,
  category VARCHAR(32) NOT NULL,
  payload VARCHAR(512) NOT NULL,
  amount DECIMAL(14,2) NOT NULL,
  batch_num BIGINT UNSIGNED NOT NULL INVISIBLE,
  PRIMARY KEY (record_id, batch_num),
  KEY ix_event_ts (event_ts)
)
PARTITION BY LIST (batch_num) (
  PARTITION p_seed VALUES IN (0)
);
  1. Create perf_t_001 through perf_t_100 for the 10–100-file fan-out test. Use a script to generate the DDL, but save the generated SQL in $TEST_ROOT/$RUN_ID/sql/.
  2. Create the Resource Mapping before uploading any object. In the UI Registered Table/Resource Mappings page, add one mapping per target with the exact test compartment display name, bucket name, and a unique pattern such as performance/<run-id>/perf_t_001/*.csv; map it to perf_hw.perf_t_001. Export or record the mapping IDs.
  3. Verify the mapping gate: confirm the mapping list shows the exact compartment, bucket, resource-name pattern, target database, and target table. The pattern must match the complete Object Storage object name (including the run prefix and .csv suffix). Do not start an upload when the list is empty or the pattern differs; the Function will report “No Resource Mappings entry matches this compartment, bucket, and resource name.”
  4. For each additional table or file group, create and verify a separate mapping before its first upload. Keep the Event Rule prefix and mapping pattern aligned; the Event Rule filters delivery, while the mapping selects the destination table.
  5. Confirm the mapping is visible in the UI and that fndb.object_storage_mappings has the expected target. Run one 10 KB smoke file before starting a matrix.
  6. For repeatability, reset only test data: delete the run prefix, truncate/retire its test partitions, and recreate the test schema if necessary. Do not manually delete control rows needed for audit analysis until results are archived.

6. Generate test data and a machine-readable manifest

Generate deterministic CSVs with a manifest containing object name, target table, event type, byte size, row count, checksum, business-key range, upload time, and expected outcome. Keep a fixed row shape for direct comparisons; add a separate wide-row variant if production rows are materially wider.

# For each requested byte size, generate rows until the CSV is at or just above the target size.
# Use a unique, non-overlapping record_id range for every source object.
python3.13 tools/generate_perf_csv.py \
  --output "$TEST_ROOT/$RUN_ID/input/perf_t_001/small-10k.csv" \
  --target-bytes 10240 --record-id-start 1000000 --seed 101 \
  --manifest "$TEST_ROOT/$RUN_ID/manifest/objects.ndjson"

sha256sum "$TEST_ROOT/$RUN_ID/input/perf_t_001/small-10k.csv" \
  >> "$TEST_ROOT/$RUN_ID/manifest/checksums.txt"

If a generator utility is not yet available, implement it with Python's standard-library csv and random.Random(seed). It should write the header once, generate deterministic record_id,event_ts,category,payload,amount values, stop at the requested size boundary, then append one NDJSON manifest record. The manifest is the authoritative expected-result dataset.

Data integrity control: allocate non-overlapping business-key ranges per source file. For every run, compare expected and loaded row counts; sample keys and verify the active batch registry. This prevents a fast but duplicate-producing run from being reported as a success.

7. Mapping preflight gate

Before Scenario A, stop and verify routing in the UI. The required order is: create target table → create Resource Mapping → verify mapping row → upload object. An Event Rule alone is not a table mapping.

  1. Open Resource Mappings and create the mapping for the exact run prefix and target table.
  2. Confirm the displayed compartment and bucket equal the event payload values, and that the shell-style pattern matches the complete object name.
  3. Upload one 10 KB canary file and wait for a successful terminal transaction. If the Function reports No Resource Mappings entry matches this compartment, bucket, and resource name, do not continue: correct the mapping and rerun the canary.
  4. Record the mapping ID, target table, canary object name, and terminal event ID in the manifest.

8. Put objects in Object Storage and run Scenario A — file-size scaling

Run one create event at a time against a reset target. This establishes the uncontentious cost curve. Repeat the same matrix for update events after an initial baseline file has been successfully loaded.

LOCAL_FILE="$TEST_ROOT/$RUN_ID/input/perf_t_001/small-10k.csv"
OBJECT_NAME="$PERF_PREFIX/perf_t_001/small-10k.csv"
UPLOAD_END_UTC=$(date -u +%Y-%m-%dT%H:%M:%S.%3NZ)
oci os object put --namespace "$OBJECT_STORAGE_NAMESPACE" --bucket-name "$PERF_BUCKET" \
  --name "$OBJECT_NAME" --file "$LOCAL_FILE" --force \
  > "$TEST_ROOT/$RUN_ID/logs/object-put-small-10k.json"
printf '%s,%s,%s\n' "$OBJECT_NAME" CREATE "$UPLOAD_END_UTC" \
  >> "$TEST_ROOT/$RUN_ID/manifest/uploads.csv"
StepProcedureMeasurements and pass criteria
1. PrepareCreate an approved target table and mapping; generate the selected file-size band with known rows and checksum.Record configuration and baseline DB/Function metrics.
2. CreateUpload one CSV and wait for a terminal SUCCESS or ERROR transaction.End-to-end time, phase times, rows/s, MB/s, peak temporary disk, Function duration, and target correctness.
3. UpdateReplace the same source object with a new, disjoint version of the same file size.Replacement latency; correct active batch; retirement of the prior batch per policy; no residual staging table.
4. ResetDelete the object and verify the associated target partition is removed; return the test target to its baseline state.Delete latency, partition-removal result, audit record, and absence of unexpected rows.
For each size: throughput_MBps = file_bytes / 1,048,576 / phase_seconds.   Overhead_seconds = median(end_to_end_seconds) − median(file_bytes / measured_bulk_MBps).

Plot file size on a logarithmic x-axis against median/p95 end-to-end latency and MB/s. The point where throughput stops increasing materially identifies a practical file-size range; the small-file curve quantifies fixed per-event overhead.

8. Scenario B — medium and large files (100 MB–1 GB)

Use 100 MB, 250 MB, 500 MB, 750 MB, and 1 GB files. First run one event at a time; then run the best two sizes at two, four, and the configured maximum safe concurrent invocations. Stop a load level if it breaches a guardrail.

GuardrailExample stop / investigate condition
Function executionTimeout, out-of-memory termination, or ephemeral disk usage above 80% of the available allocation.
Database stabilityConnection failures, sustained CPU/I/O saturation, lock waits that materially grow with each run, or failed partition exchange.
Data correctnessRow-count/checksum mismatch, duplicate business key, wrong batch number, or a staging table remaining after a successful event.
Service levelp95 completion time exceeds the agreed target or error/retry rate exceeds the agreed threshold.

Also record the time to analytic readiness in HeatWave, if that publication step is asynchronous or separately triggered. Report ingestion completion and HeatWave availability as separate service-level measures.

9. Scenario C — many small files across target tables

Use 10 KB files with unique source names and disjoint key ranges. Each file maps to a different target table, so this scenario measures event delivery, Function concurrency, database connection use, and control-table contention without serializing on one business table.

For each run calculate: accepted objects/minute, successful objects/minute, p50/p95/p99 upload-to-success latency, concurrent invocations, database connections, and retry/error count. Repeat the 50- and 100-file tests with a steady ramp (one upload every 1–5 seconds) to distinguish a burst limit from a sustainable rate.

10. Scenario D — mixed create, update, and delete events

Prepare a stable population of mapped files, then issue a mixed workload. Maintain per-object sequencing: do not intentionally move records between two files in this scenario because multi-file changes are not atomic. Test record migration separately using the documented remove-first, add-second procedure.

11. Measure event latency and derive performance projections

Record the local upload-complete timestamp before polling. Then query the control schema by the exact bucket, object name, target, and action until a terminal event status is present. Store the raw database timestamp and the polling timestamp; account for clock offset measured at run start.

mysql --defaults-extra-file=/path/to/test-client.cnf --batch --skip-column-names -e "
SELECT event_status, batch_num, created_at, message
FROM fndb.event_tx_log
WHERE bucket_name = '$PERF_BUCKET'
  AND resource_name = '$OBJECT_NAME'
  AND event_action = 'CREATE'
ORDER BY id DESC LIMIT 1;"

Calculate upload-to-terminal latency as terminal transaction time minus upload-complete time after normalizing clocks. Keep separate fields for OCI Event delivery delay, Function start delay, download/load/exchange duration, and HeatWave-ready time where instrumented. Poll at a fixed interval (for example one second); preserve each poll result so a timeout is distinguishable from an absent event.

Derive projections from stable measured points, separately for small and bulk files. Do not extrapolate one 10 KB result directly to 1 GB, or a single-file result directly to a 100-event burst.

Small-file completion estimate = fixed_event_overhead + (file_MB / measured_small_file_MBps)
Bulk-file completion estimate = fixed_bulk_overhead + (file_MB / measured_bulk_MBps)
Sustainable objects/minute = min(measured_completion_rate, Function concurrency × 60 / p95_duration_s, DB-safe completion rate)
Temporary storage requirement = concurrent_bulk_files × max_file_size × download_multiplier × safety_factor
Use download_multiplier ≥ 1 and safety_factor ≥ 1.25; set them from observed peak disk use rather than assumptions.

Report uncertainty: present a forecast range using median and p95 measurements, state the tested configuration, and apply a safety margin before turning measured throughput into a production commitment.

12. Run sheet and final performance report

  1. Capture environment/configuration snapshot and verify empty or reset test targets.
  2. Generate the data manifest and upload files using a controlled driver with timestamps.
  3. Collect OCI event/Function logs, control-table records, MySQL metrics, HeatWave availability, and target checks after every run.
  4. Classify every non-success outcome: event delivery, mapping, download, parse, database connection, load, exchange, cleanup, or HeatWave publication.
  5. Merge the object manifest, upload timestamps, control-table terminal records, Function phase logs, and database/HeatWave metrics into a results CSV or parquet dataset keyed by run_id and object name.
  6. Create the final HTML/PDF report with: environment snapshot; scenario matrix; raw-result location; p50/p95/p99 latency by action and size; MB/s; objects/minute; disk peak; DB resource pressure; error/retry rate; correctness results; capacity projections; and recommended operating envelope/alerts.
Acceptance criteriaDefine before testing
Example: zero correctness failures; p95 under the chosen service-level target; disk below 80%; and no sustained DB saturation.
DeliverablesInput manifest, raw metrics/log references, run results, charts, configuration snapshot, failures, capacity projection, and recommended alert thresholds.
