Executive result

Outcome: synchronous ingestion passed at 10, 100, and 500 MiB. Detached ingestion passed at 1, 2, and 5 GiB. Every case validated the exact row count, completed partition exchange, deleted its source object, truncated the mapped partition, and left no staging table.

  • The wide-row workload sustained approximately 14.9 MiB/s from 500 MiB through 5 GiB. A tenfold increase from 500 MiB to 5 GiB increased processing time by 10.26×, showing near-linear scaling.
  • A controlled 500 MiB narrow-row comparison closely matched the prior row count: 8,096,895 rows now versus 8,080,750 previously. Processing improved from 117.454 to 108.640 seconds: 7.50% less time and 8.09% more MiB/s.
  • The 1.3 TB allocation is consistent with higher usable storage activity, but storage allocation is not isolated as the sole cause. Database load, cache state, object delivery, row width, batching, and monitoring aggregation also affect results.

Method

  1. Reset only mapping 1 and fntestdb.perf_t_001; create the target, seed partition, UI profile, and exact resource mapping.
  2. Generate deterministic CSV with valid record_id,event_ts,category,payload,amount columns. Primary rows used a 480-byte payload and averaged about 533 bytes per row.
  3. Upload one unique object at a time through OCI CLI multipart upload. Use sync through 500 MiB, then dynamically switch the mapping to detached for 1–5 GiB.
  4. Measure delivery latency from CloudEvent time to Function receipt and processing duration from Function receipt to its terminal transaction record.
  5. Verify exact rows, terminal success, immutable mode snapshot, and zero staging residue. Delete the object and wait for successful partition cleanup before advancing.
  6. Run an additional 500 MiB comparison with a 12-byte payload, averaging 64.75 bytes per row, to match the prior 50 GB-storage workload.

What the plateau means

  • At 14:30–14:34, database receive averaged 16.765 million bytes per reported sample, or 15.989 MiB if the statistic is bytes/second. That is about 7.2% above the measured 14.911 MiB/s CSV rate, a plausible protocol and SQL overhead relationship.
  • This correlation establishes a steady end-to-end pipeline rate. It does not prove the database NIC or OCI network is maxed out. A network limit requires the metric aggregation, shape bandwidth limit, and a concurrency test that attempts to push above the plateau.
  • Database transmit reached substantially higher values than receive. That traffic cannot safely be attributed only to Function responses; it may include MySQL service, replication, or internal traffic represented by the metric.
  • CPU remained mostly well below saturation while disk operations and network traffic increased. The result is consistent with an I/O/pipeline-bound workload rather than a CPU-bound MySQL.8 workload.

Execution-mode logging defect found and fixed

During the run, the UI relabeled historical rows from the mapping’s current mode and showed a newly submitted detached event as sync until its worker completed. The database did not previously store an immutable execution-mode snapshot.

  • Added nullable invocation_mode snapshots to object_event and event_tx_log.
  • The parent Function records the resolved mode immediately; detached self-invocation carries the snapshot to the worker; terminal transaction rows persist it.
  • UI Event TX, Object Storage Event, Registered Table, Recent TX, Audit Log, and Detached Process queries now use event snapshots rather than mutable mapping state.
  • Known VM 6 performance rows were safely backfilled from their unambiguous test object names: 10 sync and 2 detached records before the fix. A post-deployment canary persisted DETACHED on both create and delete in both tables.
  • Legacy records without reliable evidence display UNKNOWN, avoiding a false mode assignment.
